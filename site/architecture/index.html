Kiến trúc htn-tunnel

01 Vấn đề htn-tunnel giải quyết

Bạn đang dev web app trên localhost:3000. Khách hàng muốn xem demo. Bạn không thể gửi link localhost:3000 cho họ — vì máy bạn sau NAT, không có IP public, internet không gọi vào được.

3 cách truyền thống — và vấn đề của chúng

CáchVấn đề
Deploy lên server thật Mỗi lần sửa code phải deploy lại — chậm, phức tạp
Port forwarding trên router Cần quyền admin, IP động hay đổi, không có HTTPS
VPN Khách hàng phải cài client — overkill cho 1 URL demo

Tunnel giải quyết cả 3

  • Không cần deploy — code vẫn chạy local, mọi thay đổi thấy ngay
  • Không cần router — client kết nối ngược ra server, vượt qua NAT
  • Không cần cài gì — người xem chỉ cần mở URL HTTPS bình thường
💡
Analogy đời thường

Giống như bạn đang ở trong toà nhà không có địa chỉ (sau NAT). Thay vì báo địa chỉ cho khách, bạn thuê một văn phòng ảo (tunnel server) và tạo đường dây điện thoại từ trong toà nhà ra văn phòng đó. Khách gọi tới văn phòng, văn phòng chuyển tiếp vào đường dây cho bạn.

02 Bức tranh tổng thể

 ┌────────┐         ┌─────────────┐         ┌────────────┐
 │Browser │────────▶│Tunnel Server│◀────────│  Máy dev   │
 │(khách) │  HTTPS  │   (VPS)     │  Kết nối│   (bạn)    │
 │        │         │             │   ngược │            │
 └────────┘         └─────────────┘         └─────┬──────┘
                                                   │
                                                   ▼
                                             localhost:3000
                                              (app của bạn)

3 thành phần:

  1. Browser của khách — gửi HTTPS request bình thường, không cần biết gì
  2. Tunnel Server trên VPS — điểm trung gian có IP public, nhận và chuyển tiếp
  3. Tunnel Client trên máy dev — giữ kết nối với server, forward tới localhost
📌
Điều đặc biệt: Client → Server, không phải ngược lại

Máy dev thường sau NAT/firewall, server không gọi vào được. Nên client phải chủ động kết nối ra ngoài rồi "giữ đường dây" để server gửi traffic ngược vào. Toàn bộ kiến trúc xoay quanh nguyên tắc này.

02b Luồng hoạt động

Bạn đã biết các thành phần. Giờ hãy theo dõi gói tin di chuyển qua hệ thống — mỗi diagram bên dưới cho thấy một luồng cụ thể, với các packet chạy theo thời gian thực.

Luồng 1: Toàn cảnh một HTTPS request

Từ lúc browser gõ URL đến khi nhận được response — qua 4 hop.

Luồng 2: Client kết nối server lần đầu

TLS handshake → xác thực token → nâng cấp lên yamux → mở control stream.

Luồng 3: HTTP request đi qua yamux

SNI Router phân loại request, yamux mở stream mới, client forward xuống localhost.

Luồng 4: Cấp wildcard cert qua DNS-01

Server xin cert *.tunnel.com — Let's Encrypt yêu cầu TXT record, Cloudflare tạo tự động.

Luồng 5: WebSocket upgrade

HTTP Upgrade → 101 Switching Protocols → raw bytes hai chiều qua toàn bộ tunnel.

03 Tầng 1: Kết nối ngược (Reverse connection)

Vai trò: Tạo đường truyền từ máy dev ra internet mà không cần port forwarding.

Client mở 1 TCP connection ra server, giữ kết nối đó mãi. Khi có request đến, server gửi data qua kết nối đã có — không cần mở kết nối mới.

Công nghệ: TCP + TLS

Công nghệVai tròLý do chọn
TCP Kết nối 2 chiều, giữ lâu dài Mọi firewall đều cho qua port 443; kết nối ổn định, thứ tự byte được đảm bảo
TLS Mã hoá toàn bộ dữ liệu Không ai nghe trộm được; browser bắt buộc HTTPS; port 443 ít bị chặn nhất

Tại sao không dùng HTTP thuần?

HTTP mỗi request là 1 round-trip request/response. Không phù hợp cho tunnel vì server cần chủ động push data khi có request mới, trong khi HTTP client mới là bên phải hỏi trước.

Tại sao không dùng WebSocket?

WebSocket cũng được, nhưng chỉ cho 1 "đường dây" mỗi kết nối. Muốn nhiều request song song → cần multiplexing → giải pháp ở tầng sau.

📞
Analogy

Client gọi điện cho server rồi giữ máy suốt ngày. Khi server muốn nói gì, chỉ việc nói vào đường dây đã mở sẵn — không phải gọi lại từ đầu.

04 Tầng 2: Multiplexing — Nhiều stream trên 1 kết nối

Vấn đề: Nếu chỉ có 1 đường dây và 10 requests đến cùng lúc, chúng phải xếp hàng từng cái — chậm khủng khiếp (gọi là head-of-line blocking).

Giải pháp: Multiplexing — chia đường dây thành nhiều "stream" ảo song song. 10 requests → 10 streams độc lập, không chờ nhau.

Công nghệ: Yamux

Library multiplexing của HashiCorp (cùng công ty làm Terraform, Consul, Vault). Implement dạng TCP-in-TCP: 1 TCP thật, nhiều stream ảo bên trong, mỗi stream có flow control độc lập.

Tại sao chọn Yamux mà không phải HTTP/2?

Tiêu chíHTTP/2Yamux
Thiết kế cho HTTP request/response Generic byte stream
Complexity Cao (flow control, HPACK, priority) Thấp — chỉ multiplexing
Forward TCP thuần Phải đóng gói vào HTTP format Tự nhiên, không cần wrapper
Debug Khó (binary, cần tool) Dễ hơn

htn-tunnel cần multiplex cả HTTP streams lẫn TCP stream thuần (database, SSH). Yamux tự nhiên hơn vì không ràng buộc bởi HTTP semantic.

📦
Analogy

Yamux giống hệ thống bưu điện — 1 xe tải (TCP) chở nhiều bưu kiện (stream), mỗi bưu kiện có địa chỉ riêng. HTTP/2 giống xe chỉ chở được bì thư có tem (HTTP format), không chở được hàng thường.

05 Tầng 3: Bảo mật — TLS + Wildcard certificate

Vai trò: HTTPS cho mọi subdomain — https://anything.tunnel.com

Vấn đề khó: cert phải là wildcard *.tunnel.com, không phải từng subdomain cụ thể — vì user có thể tạo bất kỳ subdomain nào.

Công nghệ 1: Let's Encrypt — CA miễn phí, tự động

Let's Encrypt là Certificate Authority (CA) miễn phí, được trust bởi mọi browser, dùng chuẩn ACME protocol. Cert có hạn 90 ngày, cần auto-renew.

Để xin wildcard cert, Let's Encrypt cần bạn chứng minh sở hữu domain qua DNS-01 challenge — tạo 1 TXT record cụ thể trong DNS. Đây là cách duy nhất cho wildcard; cách kia (HTTP-01) chỉ cấp cho 1 domain cố định.

Công nghệ 2: Cloudflare — DNS có API programmatic

Để tạo TXT record tự động (không tay), cần DNS provider có API. Cloudflare cung cấp API ổn định, free tier đủ dùng.

Quy trình xin cert tự động

Server → Let's Encrypt: "Cho tôi cert *.tunnel.example.com"
Let's Encrypt: "Tạo TXT record _acme-challenge = ABC123"
Server → Cloudflare API: Tạo TXT record
Cloudflare: "OK, đã tạo"
Server → Let's Encrypt: "Check đi"
Let's Encrypt → DNS query: TXT record đúng rồi ✓
Let's Encrypt → Server: "Đây cert, hiệu lực 90 ngày"
Server → Cloudflare API: Xoá TXT record (cleanup)

Công nghệ 3: certmagic — quản lý cert lifecycle

Library của Caddy web server. Tự động handle: xin cert lần đầu, renew 30 ngày trước khi hết hạn, retry nếu fail. Server không cần restart để renew cert.

🌐
Nếu domain không ở Cloudflare?

Cloudflare cho phép dùng nó như "nameserver proxy". Domain vẫn thuộc bạn ở registrar gốc, nhưng nameserver trỏ qua Cloudflare. Từ đó Cloudflare quản lý DNS records và bạn có API để auto-renew cert. Nhiều provider VN không có API — đây là giải pháp.

Trade-offs

Điểm mạnhĐiểm yếu
Miễn phí, tự động hoàn toàn Bắt buộc phụ thuộc vào DNS provider có API
Wildcard → mọi subdomain đều HTTPS ngay htn-tunnel hiện chỉ support Cloudflare
certmagic xử lý toàn bộ, không cần viết code ACME Nếu Cloudflare API lỗi → không renew được
🏠
Analogy

Let's Encrypt là cục hộ khẩu. DNS-01 là cách chứng minh nhà của bạn bằng cách treo biển ở cửa. Cloudflare là người giúp treo biển tự động. certmagic là người quản lý nhắc bạn gia hạn hộ khẩu mỗi 90 ngày.

06 Tầng 4: Định tuyến — SNI và subdomain

Vấn đề: Server nhận HTTPS trên port 443. Cả alice.tunnel.com và bob.tunnel.com đều đến cùng IP, cùng port. Làm sao biết request cho client nào?

Công nghệ: SNI (Server Name Indication)

SNI là phần mở rộng của TLS — khi bắt tay, browser gửi trước tên domain mà nó muốn kết nối. Server đọc SNI và biết request cho domain nào trước khi decrypt toàn bộ data. 

Browser → "Hello, tôi muốn nói chuyện với alice.tunnel.com"
           [SNI field trong TLS ClientHello — plaintext, chưa encrypt]
Server  → Đọc SNI → Biết là alice → Lookup → Forward tới client Alice

Tại sao SNI mà không dùng HTTP Host header?

HTTP Host header cũng chứa domain, nhưng nằm trong HTTP request — phải hoàn tất TLS handshake, decrypt, parse headers mới biết. SNI biết ngay lập tức ở tầng TLS, nhanh và sạch hơn. Cũng có thể dùng Host làm fallback nếu SNI rỗng (hiếm gặp).

Sau khi có subdomain

  1. Lookup trong map subdomain → tunnel client — O(1)
  2. Mở 1 yamux stream tới client đó
  3. Forward HTTP request qua stream, đợi response, trả về browser
  4. Nếu subdomain không tồn tại → trả 502 Bad Gateway
✉️
Analogy

SNI giống địa chỉ ghi ngoài bì thư. Bưu điện không cần mở thư ra mới biết gửi đâu — chỉ đọc bì thư là đủ. Nhanh, không cần "mở" gì.

07 Tầng 5: Giao thức điều khiển

Vai trò: Client và server trao đổi thông tin vận hành — đăng nhập, đăng ký tunnel, giữ kết nối sống, gửi log.

Công nghệ: JSON over framed TCP

Frame format đơn giản:

┌──────────────┬──────────────┬──────────────────────┐
│ Length (4B)  │  Type (1B)   │   JSON Payload       │
│  Big-Endian  │  0x01–0x10   │     N bytes          │
└──────────────┴──────────────┴──────────────────────┘

Các loại message

MessageHướngMục đích
Auth Client → Server Gửi token để đăng nhập
AuthResponse Server → Client Xác nhận hoặc từ chối
TunnelRequest Client → Server "Tôi muốn tunnel cho localhost:3000"
TunnelResponse Server → Client Trả URL public đã được cấp
Heartbeat / Ack Cả 2 chiều Ping-pong giữ kết nối sống mỗi 30s
RequestLog Server → Client Log request để hiện trên dashboard client

Tại sao JSON mà không phải Protobuf?

Tiêu chíJSONProtobuf
Performance Chậm hơn 2–3x Nhanh hơn
Debug bằng mắt Đọc trực tiếp được Cần tool decode
Schema Không bắt buộc Bắt buộc compile .proto
Volume message Vài message/tunnel/ngày — không phải bottleneck

Control plane chỉ có auth, tunnel request, heartbeat 30s/lần — tần suất thấp. Tốc độ không quan trọng bằng khả năng debug. JSON đổi lại đọc được bằng mắt — đáng giá.

📌
Control tách biệt khỏi Data

Control messages chạy trên control stream — stream đầu tiên trong yamux session. HTTP requests chạy trên các data streams riêng biệt. Tách biệt này giúp heartbeat và auth không bao giờ bị nghẽn vì data traffic nặng.

08 Tầng 6: WebSocket — Trường hợp đặc biệt

Vấn đề: WebSocket bắt đầu như HTTP request bình thường, nhưng sau khi server trả về 101 Switching Protocols, connection chuyển thành raw bidirectional bytes — không còn là HTTP nữa.

Reverse proxy HTTP thông thường không hoạt động với WebSocket vì nó chờ response xong rồi mới đóng connection. WebSocket không bao giờ "xong".

Giải pháp: Hijack connection

  1. Detect header Upgrade: websocket
  2. Server "hijack" TCP connection từ HTTP handler — lấy raw bytes
  3. Mở yamux stream mới tới client
  4. Forward HTTP upgrade request qua stream, đọc response 101 ngược về
  5. Bắt đầu pipe raw bytes 2 chiều — từ đây server chỉ relay
⚠️
Bug thực tế đã xảy ra — bài học quan trọng

Code đầu tiên dùng bufio.Reader để parse HTTP response. bufio có internal buffer 4KB — nó đọc nhiều hơn chỉ là HTTP headers, có khi đã nuốt cả WebSocket frame đầu tiên từ backend vào buffer.

Kết quả: client nhận được response 101 nhưng không nhận data đầu tiên — Socket.IO fail handshake, rất khó reproduce và debug.

Fix: sau khi parse response, flush phần còn buffered về client trước khi bắt đầu raw proxy. Bài học: trộn HTTP parsing với raw stream là cạm bẫy. Luôn kiểm tra buffer còn gì không.

Công nghệ dùng

  • http.Hijacker — Go stdlib, cho phép lấy raw TCP connection từ HTTP server
  • io.Copy — copy bytes 2 chiều giữa 2 connection

Không có library đặc biệt — chỉ là careful byte handling.

09 Tầng 7: Xác thực — 3 cấp độ

Vai trò: Kiểm soát ai được dùng hệ thống, với quyền gì, và ngăn chặn lạm dụng.

Cấp 1: Anonymous — thử không cần đăng ký

Token rỗng = anonymous mode. Server gán token ảo anon:{IP}. Mục đích: cho người dùng thử nhanh, không barrier.

Cấp 2: API Key — người dùng đăng ký

Format: htk_ + 32 ký tự random. Lưu trong JSON file trên server. Mỗi key gắn với: tên chủ sở hữu, danh sách subdomain được phép claim, số tunnel tối đa.

Cấp 3: Admin token — quản trị viên

Token duy nhất cho admin dashboard, set qua config YAML của server. Cho phép xem tất cả users, kill tunnel, sửa config.

So sánh quyền

QuyềnAnonymousAPI Key
HTTP tunnel Có (random subdomain) Có (custom subdomain)
Custom subdomain Không Có
TCP tunnel Không Có
Số tunnel đồng thời 1 per IP Theo config
Thời gian tồn tại 2 giờ Vĩnh viễn (khi online)

Rate limit — 3 tầng

  1. Pre-auth IP limit — 10 kết nối/phút/IP. Chặn DDoS trước khi đọc bất kỳ byte nào.
  2. Per-token request limit — số request/phút cho mỗi tunnel.
  3. Tunnel count limit — số tunnel đồng thời cho mỗi API key.

Tại sao không dùng OAuth / JWT?

Overkill. API key đơn giản hơn nhiều: không cần OAuth server, không cần refresh token, lookup O(1) bằng hash map, user chỉ cần copy/paste 1 chuỗi. Trade-off: nếu key bị leak phải revoke + tạo mới — với self-hosted tunnel, chấp nhận được.

Tại sao lưu JSON file mà không phải database?

Số user ít (vài chục đến vài nghìn), reads nhiều hơn writes nhiều lần, backup = copy file. Nếu scale lên hàng chục nghìn user → chuyển SQLite hoặc PostgreSQL. Hiện tại database là overkill thực sự.

10 Tầng 8: Dashboard thời gian thực

Yêu cầu: User muốn xem request log live — thấy request đến ngay khi xảy ra, không phải F5 mỗi vài giây.

Công nghệ: SSE (Server-Sent Events)

SSE = HTTP response không bao giờ đóng. Server push event mới xuống bất cứ lúc nào, browser nhận tức thì.

Tại sao SSE mà không WebSocket?

Tiêu chíSSEWebSocket
Chiều truyền Server → Client (đủ cho dashboard) 2 chiều (không cần thiết)
Auto-reconnect Built-in trong browser Phải tự code
Qua proxy/firewall Chuẩn HTTP, không vấn đề Đôi khi bị chặn
Complexity Đơn giản hơn Phức tạp hơn

Data source: Ring buffer

Server giữ 10.000 log entries gần nhất trong RAM. Request mới → append, drop entry cũ nhất nếu full. Query: recent N entries, thống kê theo phút, top paths.

Tại sao không dùng database cho log?

Dashboard chỉ cần xem live và 30 phút gần nhất. Ring buffer đủ, nhanh, không cần install gì. Trade-off: mất data khi restart server — chấp nhận được cho monitoring real-time. Nếu cần lưu vĩnh viễn → ghi vào file log hoặc push ra Prometheus.

⚠️
SSE qua nginx cần header đặc biệt

Nginx mặc định buffer response — SSE events sẽ bị giữ lại cho đến khi đầy buffer mới flush, khiến user thấy log bị delay hàng chục giây. Fix: thêm header X-Accel-Buffering: no vào SSE response. Không có header này, SSE qua nginx trông như bị hỏng.

12 Bài học rút ra

Nếu muốn tự build tunnel tương tự, đây là những điều quan trọng nhất:

  1. Đừng tự viết multiplexing. Dùng yamux, HTTP/2, hoặc QUIC. Multiplexing đúng cách có nhiều edge case (flow control, stream lifecycle, window update). Library đã giải quyết hết.
  2. Client phải chủ động kết nối ra. Đừng cố mở port trên máy dev. Đường truyền luôn từ trong ra ngoài — qua NAT tự nhiên.
  3. DNS-01 wildcard cert là bắt buộc. Không có cách khác cho wildcard. Chấp nhận dependency vào DNS provider có API từ đầu.
  4. SNI là bạn, không phải Host header. Routing ở tầng TLS nhanh hơn, sạch hơn, không cần đợi decrypt HTTP.
  5. WebSocket = careful byte handling. Đừng trộn HTTP parser với raw stream. Luôn check buffered data còn lại sau khi parse xong HTTP response.
  6. Không cần database từ đầu. Ring buffer + JSON file đủ cho hàng nghìn user. Thêm DB khi thực sự cần.
  7. SSE đơn giản hơn WebSocket cho server push. Auto-reconnect built-in, qua proxy tốt hơn, code ít hơn.
  8. Ngôn ngữ quan trọng. Go phù hợp đặc biệt với pattern này: goroutines cho mỗi connection, channels cho pub/sub, stdlib mạnh về network, static binary dễ deploy.
  9. Self-hosted cần UX tốt cho setup. Viết wizard init tự làm mọi thứ (DNS check, config, systemd, cert). Nếu user phải gõ 10 lệnh tay → họ sẽ làm sai.
  10. Document cái WHY. Code nói WHAT. Comment và doc nói WHY. Người đọc 3 tháng sau — kể cả bạn — sẽ cảm ơn.
