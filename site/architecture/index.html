01 Tổng quan

Bài toán cốt lõi của tunnel là đơn giản: bạn đang phát triển một web app trên localhost:3000 và cần share link cho khách hàng, teammate, hoặc webhook của bên thứ ba gọi vào. Vấn đề là máy bạn nằm sau NAT — không có IP public, không có port forwarding, không thể nhận kết nối từ bên ngoài.

Có ba cách tiếp cận phổ biến:

  • SSH reverse tunnel (ssh -R 8080:localhost:3000 user@vps) — đơn giản, có sẵn, nhưng không HTTP-friendly: không có TLS, không có custom domain, không parse được HTTP headers để routing.
  • VPN (WireGuard, Tailscale) — mạnh nhưng overkill cho use case này. VPN kết nối mạng, không per-port. Phải cấu hình phức tạp hơn.
  • Reverse proxy tunnel (ngrok, cloudflared, htn-tunnel) — chuyên biệt cho bài toán này. HTTPS endpoint ngay lập tức, custom subdomain, dashboard analytics.

htn-tunnel thuộc nhóm thứ ba, nhưng là self-hosted — bạn chạy server trên VPS của mình, toàn quyền kiểm soát. Ba yêu cầu cốt lõi quyết định toàn bộ kiến trúc:

  1. Client-initiated connection — client (máy dev của bạn) phải là bên chủ động kết nối đến server, vì client sau NAT không thể nhận inbound connection. Server không bao giờ dial ngược lại client.
  2. Multiplexing — mỗi HTTP request từ browser phải được forward song song mà không cần mở lại TCP connection mới (quá chậm). Cần một cơ chế ghép nhiều logical stream trên cùng một TCP connection.
  3. Public HTTPS endpoint — browser hiện đại bắt buộc HTTPS. Cần wildcard TLS cert cho *.tunnel.example.com để routing theo subdomain hoạt động.

Toàn bộ kiến trúc htn-tunnel xoay quanh ba ràng buộc này. Mỗi quyết định thiết kế đều có thể trace ngược về một trong ba yêu cầu trên.

Port 443 nhận HTTPS từ browser — SNI extractor đọc server name từ TLS ClientHello để biết subdomain nào được yêu cầu, không cần decrypt toàn bộ payload. Port 4443 là control port — client kết nối vào đây, xác thực, rồi duy trì yamux session. Port 1807 là dashboard web — API và SSE stream cho admin UI.

03 Tầng 1: Control Plane

Control plane là bước đầu tiên khi client kết nối — thiết lập session, xác thực, và mở yamux trước khi bất kỳ tunnel nào được tạo.

Handshake flow từng bước

  1. Client dial TLS tới server:4443. Dùng tls.Dialer.DialContext() — ở dev mode có InsecureSkipVerify, production dùng cert hợp lệ từ Let's Encrypt.
    File: internal/client/client.go:35
  2. Gửi AuthMsg với token. Wire format: [len:4B][ver:1B=0x01][type:1B=0x01][json:{token}]. Token rỗng nếu anonymous.
  3. Server validate token — ba loại:
    • htk_* → KeyStore lookup (JSON file persist trên disk)
    • anon:{ip} → anonymous mode, giới hạn nghiêm ngặt
    • Legacy tokens trong config YAML (đang phase out)
  4. AuthResponse — success hoặc fail với message. Nếu fail, server đóng connection ngay.
  5. Upgrade sang yamux.Session — cùng một TCP conn được wrap bởi yamux library. Từ đây, cả hai bên có thể open/accept streams độc lập.
  6. Mở control stream — stream đầu tiên trong session, dùng riêng cho TunnelReq, TunnelResp, Heartbeat. Các data stream sau này là riêng biệt.
💡
Tại sao yamux thay vì mở TCP mới?

Mỗi TLS handshake tốn 1–2 RTT (~20–40ms). Nếu mỗi HTTP request phải tạo lại TCP + TLS, overhead sẽ là 40–80ms per request — không chấp nhận được. Yamux giải quyết bằng cách multiplexing nhiều logical streams trên cùng một TLS connection đã sẵn có.

04 Tầng 2: Wire Protocol

Control plane dùng một binary framing protocol đơn giản — length-prefixed frames với JSON payload. Không dùng protobuf hay msgpack để giữ code đơn giản. 

┌──────────────┬──────────────┬──────────────┬──────────────────────┐
│ Length (4B)  │ Version (1B) │  Type (1B)   │   JSON Payload       │
│  Big-Endian  │    0x01      │  0x01–0x10   │     N bytes          │
└──────────────┴──────────────┴──────────────┴──────────────────────┘

Length = version(1) + type(1) + payload(N)  — không bao gồm 4 byte length field
Max message size = 1 MB  — hard cap tránh memory exhaustion

Encoder dùng mutex để thread-safe khi nhiều goroutine ghi đồng thời vào cùng một control stream. Decoder đọc đúng length byte rồi JSON-unmarshal vào struct tương ứng theo type.

Bảng message types

Type Hex Payload Hướng Mô tả
MsgAuth 0x01 {token} C→S Xác thực client
MsgAuthResponse 0x02 {success, message} S→C Kết quả xác thực
MsgTunnelReq 0x03 {type, subdomain?, local_port} C→S Yêu cầu tạo tunnel
MsgTunnelResp 0x04 {success, url, remote_port?} S→C URL tunnel được cấp
MsgHeartbeat 0x05 null C→S Keepalive ping
MsgHeartbeatAck 0x06 null S→C Keepalive pong
MsgRequestLog 0x10 {method, path, status, duration, size} S→C Request log cho dashboard
⚖️
JSON vs protobuf?

Protobuf nhanh hơn 2–3× và nhỏ hơn ~40%. Nhưng cần schema compilation, code generation, và toolchain phức tạp hơn. Với control plane — ít message, không nằm trên critical path của data — JSON đơn giản hơn và debuggable hơn. Trade-off đúng chỗ.

05 Tầng 3: Yamux Multiplexing

Yamux (Yet Another Multiplexer) của HashiCorp là thư viện multiplexing TCP streams. Concept: một TCP/TLS connection được chia thành N logical streams độc lập, mỗi stream có flow control riêng, cả hai phía đều có thể chủ động open hoặc accept streams.

Điểm khác biệt với HTTP/2: yamux không có semantic HTTP — chỉ là raw byte streams. Đây là ưu điểm khi dùng để proxy arbitrary protocols (TCP, WebSocket, gRPC raw).

Server side

// Sau khi auth thành công, wrap conn bằng yamux.Server
session, _ := yamux.Server(conn, yamuxCfg)

// Client luôn mở control stream đầu tiên
controlStream, _ := session.AcceptStream()

// Các stream sau này do server chủ động mở khi có HTTP request đến
// session.Open() → gửi yêu cầu open stream đến client

Client side

// Client wrap conn bằng yamux.Client
session, _ := yamux.Client(conn, yamuxCfg)

// Client chủ động mở control stream đầu tiên
controlStream, _ := session.Open()

// Client accept các data streams mà server mở về sau
go func() {
    for {
        stream, _ := session.AcceptStream()
        go handleDataStream(stream)
    }
}()

Config

  • KeepAlive interval: 30s — yamux-level keepalive, độc lập với application heartbeat
  • Connection write timeout: 10s — tránh goroutine bị block vĩnh viễn khi client chậm
  • Session idle timeout: 90s — nếu không có heartbeat từ application layer, đóng session
📌
Hai tầng keepalive

Yamux có keepalive riêng ở transport level. htn-tunnel thêm một lớp application heartbeat (MsgHeartbeat mỗi 30s) để detect dead sessions nhanh hơn và gửi log qua control stream. Hai tầng độc lập nhau.

06 Tầng 4: HTTP Proxy — Reverse proxy

Khi browser gửi HTTPS request đến myapp.tunnel.example.com, server phải nhận, decrypt, parse subdomain, tìm đúng client, forward request, nhận response, rồi trả về. Năm bước rõ ràng.

Bước 1: TLS termination + SNI extraction

Trước khi decrypt, server đọc SNI (Server Name Indication) từ TLS ClientHello — đây là phần plaintext trong TLS handshake, chứa hostname mà client muốn kết nối. 

tlsConn.Handshake()
subdomain := tlsConn.ConnectionState().ServerName
// "myapp.tunnel.example.com" → extract "myapp"
parts := strings.SplitN(serverName, ".", 2)
subdomain = parts[0]

Bước 2: Tunnel lookup

// O(1) map lookup — sync.RWMutex protected
ts := tunnelManager.LookupHTTP(subdomain)
if ts == nil {
    http.Error(w, "502 Tunnel not found", 502)
    return
}

Bước 3: Reverse proxy với yamuxTransport

Thay vì dùng httputil.ReverseProxy với default transport, htn-tunnel implement custom http.RoundTripper — yamuxTransport. 

func (t *yamuxTransport) RoundTrip(req *http.Request) (*http.Response, error) {
    // Mỗi HTTP request → mở 1 yamux stream mới
    stream, err := t.session.Open()

    // Ghi toàn bộ HTTP request vào stream
    req.Write(stream)

    // Đọc HTTP response từ stream
    resp, err := http.ReadResponse(bufio.NewReader(stream), req)
    return resp, err
}

Bước 4: Forwarding headers

// Set forwarding headers
req.Header.Set("X-Forwarded-For", clientIP)
req.Header.Set("X-Forwarded-Proto", "https")
req.Header.Set("X-Forwarded-Host", req.Host)

// Xóa hop-by-hop headers (Connection, Keep-Alive, Upgrade...)
// để không confuse app local
removeHopByHop(req.Header)

Bước 5: Request logging

Response writer được wrap bởi responseRecorder — capture HTTP status code và bytes written. Sau khi response hoàn tất, log entry được gửi qua control stream (MsgRequestLog) về client dashboard, đồng thời được lưu vào ring buffer server-side cho admin dashboard.

07 Tầng 5: TCP Proxy — Forward port

TCP proxy đơn giản hơn HTTP nhiều vì không cần parse — chỉ cần pipe bytes. Dùng cho database (5432), SSH (22), hoặc bất kỳ TCP service nào.

  1. tunnelManager.RegisterTCP(session) — allocate random port trong range [portMin, portMax] (config được). Retry 100 lần nếu collision.
  2. net.Listen("tcp", ":PORT") — bind port trên server.
  3. Goroutine accept connections vô hạn — mỗi incoming conn tạo một yamux stream mới.
  4. Bidirectional copy — raw bytes, không parse gì thêm: 
func proxyTCP(clientConn net.Conn, session *yamux.Session) {
    stream, _ := session.Open()
    defer stream.Close()
    defer clientConn.Close()

    done := make(chan struct{}, 2)
    go func() { io.Copy(stream, clientConn); done <- struct{}{} }()
    go func() { io.Copy(clientConn, stream); done <- struct{}{} }()
    <-done // một chiều đóng → đóng cả hai
}

08 Tầng 6: WebSocket — Xử lý đặc biệt

WebSocket là phần tricky nhất của toàn bộ codebase. Vấn đề: WebSocket bắt đầu như HTTP request, sau khi server trả 101 Switching Protocols, connection chuyển thành raw bidirectional bytes — hoàn toàn không còn là HTTP. httputil.ReverseProxy không handle được upgrade này đúng cách.

Flow xử lý WebSocket

  1. Detect header Upgrade: websocket + Connection: upgrade
  2. Hijack HTTP connection — http.Hijacker lấy raw TCP conn từ Go HTTP server
  3. Open yamux stream mới từ session của client tương ứng
  4. Rewrite Origin header — dev servers (Vite, webpack-dev-server) thường check Origin và reject nếu không match. Server rewrite về http://localhost:PORT
  5. Write HTTP upgrade request vào stream — kể cả headers đã rewrite
  6. Read upgrade response (101) từ stream
  7. Flush buffered bytes — đây là bẫy tinh vi nhất (xem callout bên dưới)
  8. Bidirectional raw proxy — từ đây là thuần bytes
// Sau khi đọc 101 response
br := bufio.NewReader(stream)
resp, _ := http.ReadResponse(br, upgradeReq)
resp.Write(clientConn)

// CRITICAL: bufio.Reader có thể đã đọc thêm bytes vào buffer
// vượt ngoài HTTP headers — phải flush về client trước
if n := br.Buffered(); n > 0 {
    buf := make([]byte, n)
    io.ReadFull(br, buf)
    clientConn.Write(buf)
}

// Bây giờ mới raw proxy
go io.Copy(stream, clientConn)
io.Copy(clientConn, stream)
⚠️
bufio.Reader trap — bug v0.4.5, fixed v0.4.6

bufio.NewReader đọc từ stream vào internal buffer theo chunk (thường 4KB). Khi parse HTTP response xong, buffer có thể chứa một phần WebSocket frame đầu tiên. Nếu không flush buffer này về client trước khi raw proxy, frame đó bị mất — WebSocket kết nối nhưng app nhận thiếu data. Bug này chỉ xảy ra khi app gửi WebSocket message ngay sau handshake, trước khi server kịp chuyển sang raw mode.

09 Tầng 7: TLS Wildcard Cert

Tunnel cần cert cho *.tunnel.example.com. HTTP-01 challenge của Let's Encrypt không support wildcard — chỉ cert cho một domain cụ thể. Giải pháp bắt buộc là DNS-01 challenge.

Flow DNS-01

  1. Server gửi Certificate Signing Request (CSR) lên Let's Encrypt
  2. Let's Encrypt yêu cầu: tạo TXT record _acme-challenge.tunnel.example.com
  3. Server dùng Cloudflare API tự động tạo DNS TXT record
  4. Let's Encrypt verify TXT record tồn tại và đúng value
  5. Let's Encrypt cấp cert — server nhận và lưu trữ
  6. Server dùng Cloudflare API xóa TXT record (cleanup)

htn-tunnel dùng thư viện certmagic của Caddy — tự handle toàn bộ lifecycle: request, store, renew (30 ngày trước expiry), rotate. Server không bao giờ cần restart để renew cert.

🌐
Tại sao chỉ Cloudflare?

certmagic hỗ trợ nhiều DNS providers (Route53, DigitalOcean, GCP, etc.) nhưng mỗi provider cần code integration riêng. htn-tunnel chỉ implement Cloudflare để giữ codebase tối giản — phù hợp với 80% self-hosters. PR welcome cho providers khác.

10 Tầng 8: Auth & Rate Limit

Ba tầng defense-in-depth — mỗi tầng chặn một loại abuse khác nhau.

Tầng 1: Pre-auth IP rate limit

10 connection attempts / phút / IP. Chặn trước khi đọc bất kỳ byte nào từ client. Dùng sliding window counter trong memory — không cần database.

Tầng 2: Auth handshake

Ba loại token, mỗi loại có permission khác nhau:

// KeyStore format — JSON file trên disk
{
  "keys": {
    "htk_abc123...": {
      "name":        "John Doe",
      "subdomains":  ["myapp", "api"],
      "max_tunnels": 10,
      "active":      true,
      "created_at":  "2026-04-23T00:00:00Z"
    }
  }
}

Validation là O(1) map lookup — keys[token].Active. File được reload khi có thay đổi (inotify / polling), không cần restart server.

11 Tầng 9: Dashboard & Analytics

Dashboard cần hai loại data: real-time stream (request log tức thì) và historical aggregates (traffic chart 30 phút). Giải pháp: SSE + in-memory ring buffer, không cần database.

SSE (Server-Sent Events) cho real-time

// Headers quan trọng để SSE hoạt động qua nginx
w.Header().Set("Content-Type", "text/event-stream")
w.Header().Set("Cache-Control", "no-cache")
w.Header().Set("X-Accel-Buffering", "no") // chặn nginx buffer SSE
w.Header().Set("Connection", "keep-alive")

Mỗi dashboard subscriber đăng ký vào pub/sub channel. Khi có request mới, server broadcast log entry đến tất cả subscribers. Subscriber chậm sẽ bị skip (non-blocking channel send) — không bao giờ làm nghẽn server.

Ring buffer — 10.000 entries

In-memory, thread-safe với sync.RWMutex. Circular buffer — khi đầy, entry cũ nhất bị overwrite. Query: recent N entries, traffic per minute, top paths, status code breakdown. Đủ cho 99% monitoring use case mà không cần setup Prometheus hay InfluxDB.

Per-minute aggregates

Goroutine background chạy mỗi phút — đọc entries từ ring buffer, aggregate thành bucket theo timestamp. Dashboard fetch 30 bucket gần nhất để vẽ bar chart. Memory footprint cố định, không tăng theo traffic.

12 Full Request Flow

Theo chân một HTTP request từ lúc browser gửi đến lúc app local nhận — tổng hợp tất cả các tầng đã phân tích.

Latency breakdown

BướcLatency ước tínhGhi chú
TLS handshake (browser → server) ~20ms TLS 1.3 = 1-RTT
SNI lookup + tunnel registry <1ms O(1) map
yamux.Open() → client accept ~RTT/2 Phụ thuộc latency server↔client
HTTP request transit → client ~RTT/2 Cùng TCP connection, không thêm handshake
Client dial localhost ~2ms Loopback, rất nhanh
App xử lý X ms Phụ thuộc app
Response transit ngược ~RTT Cùng yamux stream

Tổng overhead so với local-only: ~2×RTT (server↔client) + 20ms TLS + ~5ms misc. Với server và client cùng datacenter region, RTT ~5–10ms → overhead tổng ~30–40ms. Hoàn toàn chấp nhận được cho dev workflow.

🔁
WebSocket khác gì?

WebSocket chỉ có overhead một lần ở handshake phase (tương tự trên). Sau đó là bidirectional raw stream — latency per message chỉ là RTT server↔client, không có HTTP overhead. Thường nhanh hơn HTTP/request cho real-time use cases.

13 Key Takeaways & Gợi ý implementation

Nếu bạn muốn tự build tunneling tool tương tự, đây là những bài học quan trọng nhất:

Kiến trúc

  1. Dùng yamux hoặc HTTP/2 — đừng tự viết multiplexing. Multiplexing đúng cách (flow control, window size, backpressure) rất phức tạp. yamux của HashiCorp đã battle-tested trong Consul, Vault, Nomad.
  2. Tách biệt control plane và data plane. Control stream (JSON messages) và data streams (raw bytes) trên cùng yamux session nhưng hoàn toàn độc lập. Control không bao giờ bị nghẽn vì data và ngược lại.
  3. Client-initiated luôn luôn. Server không bao giờ dial client — client dial server. Đây là yêu cầu bắt buộc để vượt qua NAT. Kiến trúc phải phản ánh điều này từ đầu.

Implementation gotchas

  1. DNS-01 challenge là bắt buộc cho wildcard cert. HTTP-01 không support *.domain.com. Setup Cloudflare API key ngay từ đầu, đừng để sau.
  2. Test WebSocket kỹ — bufio trap là thật. Mọi HTTP parsing library đều dùng buffered reader. Khi switch sang raw mode, phải flush buffer còn lại trước. Thiếu bước này: frame loss, bug khó reproduce.
  3. Ring buffer thay database cho analytics. Đơn giản hơn 10×, không cần infra, đủ cho 99% use case. Chỉ upgrade khi có requirement retention > vài giờ hoặc multi-server.
  4. SSE cần X-Accel-Buffering: no khi đứng sau nginx. Nginx mặc định buffer response — SSE events sẽ không realtime nếu thiếu header này.

Tech stack tối giản

Thành phầnLựa chọnLý do
Runtime Go Goroutines phù hợp tự nhiên với pattern per-stream-per-goroutine. Static binary dễ deploy.
Multiplexing yamux Đơn giản, battle-tested, raw streams không tied với HTTP semantic
TLS cert certmagic Auto-renew, DNS-01 support, zero config
CLI cobra Standard Go CLI framework, subcommand support
Frontend Vanilla JS + CSS Dashboard không cần React — giảm bundle size, không cần build step

Tổng codebase htn-tunnel: ~1500 dòng logic core (không tính CLI boilerplate, config parsing, tests). Đủ để đọc hết trong một buổi chiều.
